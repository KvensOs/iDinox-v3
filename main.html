qué es iDinox

No es un bot
genérico.
Es un motor
de ligas.

Nace de la necesidad real de comunidades de HaxBall que organizaban sus ligas con planillas de Google Sheets, mensajes sueltos y mucho trabajo manual cada temporada.

Con iDinox v3 todo queda registrado desde la inscripción de un jugador hasta el palmarés histórico. La base de datos relacional garantiza que los datos de temporadas pasadas nunca desaparecen aunque el servidor arranque de cero.

La arquitectura soporta multi-modalidad nativa: X4 y X5 conviven en el mismo servidor como instancias completamente independientes, con sus propios canales, roles, equipos, temporadas y mercados.

TypeScript ESM + Discord.js v14

Tipado fuerte. Autocomplete dinámico en todos los slash commands.

SQLite + Sequelize

8 modelos relacionales. Seed automático con npx tsx. Sin servidor externo.

Multi-modalidad nativa

X4 y X5 con universos separados bajo el mismo bot y servidor.

base de datos

Ocho modelos
que lo conectan todo.

Una base relacional que guarda la historia completa de cada liga. Temporadas cerradas, equipos eliminados, palmarés de hace tres años — todo queda.

Modality

id · name (unique) · displayName
playersPerTeam · settings (JSON) · isActive

X4, X5, etc. Cada modalidad tiene canales, roles y mercado independiente. El campo settings almacena toda la configuración de Discord.

Season

id · name · modalityId (FK)
isActive · startedAt · endedAt

Temporadas independientes por modalidad. Al cerrar una, todas sus competencias se cierran en cascada automáticamente.

Team

id · name · abbreviation · logoPath
modalityId · roleId · uniformHome · uniformAway
emergencySigns · isActive

Logo descargado a disco, uniformes con validación. Soft delete: roleId = "deleted_${id}" para liberar el rol.

Player

id · discordId (unique)
username · globalName

Perfil global del jugador, completamente independiente de temporada o modalidad. Un registro por Discord ID.

Participant

nodo central
id · playerId · teamId (nullable)
seasonId · modalityId · position · isActive
UNIQUE (playerId, seasonId, modalityId)

Une Player + Team + Season. teamId = null significa agente libre. Un jugador puede tener Participant en X4 y X5 simultáneamente.

Competition

id · name · seasonId (FK)
type (league|cup|friendly|other) · isActive

Ligas, copas, amistosos. Sin modalityId propio — se hereda del Season al que pertenece.

Stat

id · participantId (FK) · competitionId (FK)
goles · asistencias · amarillas · rojas

Estadísticas granulares por Participant y competencia. Nunca se mezclan entre temporadas o modalidades.

Award

id · name · type (team|individual)
winnerId · seasonId · modalityId
competitionId (nullable) · notes

Premios individuales o por equipo. MVP, bota de oro, campeón — con notas opcionales y referencia a competencia.

// relaciones entre modelos

Modality→ Season→ Competition→ Stat← Participant← Player
Modality→ Team→ Participant· Season→ Award
Participant es el nodo central — une Player + Team + Season. El índice UNIQUE (playerId, seasonId, modalityId) garantiza integridad a nivel de base de datos sin necesidad de validaciones extra en la aplicación. Las Stats son por Participant + Competition: así el historial es granular y nunca se mezcla entre temporadas o modalidades.

arquitectura & decisiones

Bajo el capó.

Decisiones que tienen un porqué. No son caprichos — son respuestas a problemas reales de ligas que crecen y cambian cada temporada.

DT sin campo fijo en la DB

El Director Técnico no se almacena en Team. Se detecta dinámicamente: DT = quien tiene rol_dt de los settings de la modalidad y pertenece al equipo. Cambiar de DT es solo cambiar el rol de Discord — sin queries de actualización.

Soft delete en equipos

Eliminar un equipo no borra nada: isActive = false y roleId = "deleted_${id}". Libera el roleId para nuevos equipos pero preserva todo el historial de stats, participants y awards intacto.

Competition hereda modalidad de Season

Las Competition no tienen modalityId propio. Lo heredan del Season al que pertenecen. Un dato, una fuente de verdad — no hay duplicación ni posibilidad de inconsistencia entre tablas.

Participant UNIQUE a nivel DB

El índice único sobre (playerId, seasonId, modalityId) garantiza integridad sin validaciones extra en la capa de aplicación. No hay forma de duplicar un participante, ni por error ni por condición de carrera.

Cierre en cascada automático

Al cerrar una temporada, todas sus Competition activas se cierran automáticamente. Sin pasos manuales olvidados, sin competencias abiertas en temporadas archivadas que contaminen queries futuras.

Autocomplete dinámico desde la DB

Los campos de modalidad, equipo y competencia en los slash commands se cargan desde la DB en cada interacción via autocompleteModality(). Añadir una modalidad o equipo nuevo lo hace disponible al instante, sin tocar código.

Player.findOrCreate en lugar de findOne

Los DTs creados desde /league-team add pueden no haber usado nunca /start. Usar findOrCreate garantiza que siempre existe su registro de Player sin romper el flujo, aunque nunca hayan interactuado directamente con el bot.

Validación de uniformes con regex

Los campos uniformHome y uniformAway deben iniciar con /colors (regex /^\/colors\b/i). Se valida en todos los comandos que los modifican — tanto en creación como en edición posterior por el DT o sub-DT.

configuración por modalidad

El objeto ModalitySettings

Cada modalidad almacena su configuración completa en un campo JSON. El bot lee este objeto para saber a qué canales enviar notificaciones y qué roles tienen permisos de administración.

ModalitySettings — tipo TypeScript
type ModalitySettings = {
  marketOpen: boolean,
  rol_dt: string | null, // Discord role ID del DT
  rol_sub_dt: string | null, // sub-DT puede editar uniformes
  rol_admin: string | null, // admin de la modalidad
  canal_mercado_fichajes: string | null, // notificaciones de altas
  canal_mercado_bajas: string | null, // notificaciones de bajas
  canal_resultados: string | null,
  canal_logs: string | null, // discordLogger envía aquí
}

utilidades del core

Funciones que usa todo el bot

discordLogger

Envía embeds con color e icono al canal canal_logs de la modalidad. Niveles: info, success, warn, error, fatal. Todos los comandos lo usan para auditoría automática de cada acción.

Permisos granulares

isAdmin(member, client) para admin global del servidor. isModalityAdmin(member, client, settings) para admin de modalidad. DENIED_EMBED como objeto reutilizable para respuestas de acceso denegado.
