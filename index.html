qué es iDinox

No es un bot
genérico.
Es la liga
completa.

Nace de algo concreto: comunidades de HaxBall que organizaban sus ligas con planillas de Google Sheets, mensajes sueltos y mucho trabajo manual cada temporada. Datos perdidos, jugadores confundidos, admins quemados. Con iDinox v3 eso se acabó.

Todo queda registrado — desde la inscripción de un jugador hasta el palmarés histórico. La base de datos relacional garantiza que los datos de temporadas pasadas nunca desaparecen aunque el servidor arranque de cero.

La arquitectura soporta multi-modalidad nativa: X4 y X5 conviven en el mismo servidor como instancias completamente independientes, con sus propios canales, roles, equipos, temporadas y mercados.

TypeScript ESM + Discord.js v14

Tipado fuerte. Autocomplete dinámico en todos los slash commands. Código que se puede mantener.

SQLite + Sequelize

8 modelos relacionales. Backups automáticos antes de cada operación destructiva. Sin servidor externo.

Multi-modalidad nativa

X4 y X5 con universos separados bajo el mismo bot. Cada modalidad con su propia configuración completa.

Permisos granulares

isAdmin · isModalityAdmin · DT · sub-DT · estadistiquero. Verificados en tiempo real contra roles de Discord.

base de datos

Ocho modelos
que lo conectan todo.

Una base relacional que guarda la historia completa de cada liga. Temporadas cerradas, equipos eliminados, campeones de hace tres años — todo queda.

Modality

id · name (unique) · displayName
playersPerTeam · settings (JSON) · isActive

X4, X5, etc. Cada modalidad tiene canales, roles y mercado independiente. El campo settings almacena toda la configuración de Discord.

Season

id · name · modalityId (FK)
isActive · startedAt · endedAt

Solo una activa por modalidad. Al cerrar, todas sus competencias se cierran en cascada. Backup automático antes de cada cierre.

Team

id · name · abbreviation · logoPath
modalityId · roleId · uniformHome · uniformAway
emergencySigns · isActive

Logo descargado a disco, uniformes con validación. Soft delete: roleId = "deleted_${id}".

Player

id · discordId (unique)
username · globalName

Perfil global del jugador, independiente de temporada o modalidad. Un registro por Discord ID.

Participant

nodo central
id · playerId · teamId (nullable)
seasonId · modalityId · position · isActive
UNIQUE (playerId, seasonId, modalityId)

Une Player + Team + Season. teamId = null es agente libre.

Competition

id · name · seasonId (FK)
type (league|cup|friendly|other)
canalEstadisticas · isActive

Ligas, copas, amistosos. Tiene canal dedicado para cargar stats por mensaje. Hereda modalidad de su Season.

Stat

id · participantId (FK) · competitionId (FK)
goles · asistencias · cs · autogoles

Stats por Participant y competencia. Soporta negativos con clamp a 0. Nunca se mezclan entre temporadas.

Award + AwardWinner

Award: name · type (team|individual)
seasonId · competitionId (nullable) · notes
AwardWinner: playerId | teamId

Premios individuales y de equipo por temporada. Awards de equipo crean una fila por cada Participant activo.

// relaciones entre modelos

Modality→ Season→ Competition→ Stat← Participant← Player
Modality→ Team→ Participant· Season→ Award→ AwardWinner
Participant es el nodo central — une Player + Team + Season. El índice UNIQUE (playerId, seasonId, modalityId) garantiza integridad a nivel de base de datos. Las Stats son por Participant + Competition: el historial es granular y nunca se mezcla. Los AwardWinner caen en CASCADE al borrar el Award.

arquitectura & decisiones

Bajo el capó.

Decisiones que tienen un porqué. No son caprichos — son respuestas a problemas reales de ligas que crecen y cambian cada temporada.

DT sin campo fijo en la DB

El Director Técnico no se almacena en Team. Se detecta dinámicamente: DT = quien tiene rol_dt de los settings de la modalidad y pertenece al equipo. Cambiar de DT es solo cambiar el rol de Discord.

Soft delete en equipos

Eliminar un equipo no borra nada: isActive = false y roleId = "deleted_${id}". Libera el roleId para nuevos equipos pero conserva todo el historial de stats, participants y awards.

Competition hereda modalidad de Season

Las Competition no tienen modalityId propio — lo heredan del Season. Un dato, una fuente de verdad. Nunca puede haber una competencia de X4 en una temporada de X5.

Participant UNIQUE a nivel DB

El índice único sobre (playerId, seasonId, modalityId) garantiza integridad sin validaciones extra. Ni por error ni por condición de carrera se puede duplicar un participante.

Cierre en cascada automático

Al cerrar una temporada, todas sus Competition activas se cierran solas. Sin pasos manuales, sin competencias abiertas en temporadas archivadas que contaminen queries futuras.

Autocomplete dinámico desde la DB

Los campos de modalidad, equipo y competencia en los slash commands se cargan desde la DB en cada interacción. Añadir una modalidad nueva la hace disponible al instante, sin reiniciar el bot.

SEASON_SECRET + confirmar

Los comandos /season new y /season end requieren una clave secreta en .env y el nombre exacto de la temporada a cerrar. Backup automático del .sqlite antes de cualquier escritura.

Stats por canal dedicado

Cada Competition puede tener un canalEstadisticas. El evento messageCreate detecta el canal y procesa @Jugador g2 a1 cs1 automáticamente. Sin el rol correcto → ignorado silencioso.

configuración por modalidad

El objeto ModalitySettings

Cada modalidad almacena su configuración completa en un campo JSON. El bot lee este objeto para saber a qué canales enviar notificaciones y qué roles tienen permisos.

ModalitySettings — tipo TypeScript clic para copiar
type ModalitySettings = {
  marketOpen: boolean,
  rol_dt: string | null, // Discord role ID del DT
  rol_sub_dt: string | null, // sub-DT puede editar uniformes
  rol_admin: string | null, // admin de la modalidad
  rol_estadistiquero: string | null, // carga stats por canal o comando
  canal_mercado_fichajes: string | null, // notificaciones de altas
  canal_mercado_bajas: string | null, // notificaciones de bajas
  canal_resultados: string | null,
  canal_logs: string | null, // discordLogger envía aquí
}

utilidades del core

Funciones que usa todo el bot

discordLogger

Envía embeds con color e icono al canal canal_logs de la modalidad. Niveles: info, success, warn, error, fatal. Síncrono — nunca usar await logger.*. Trazabilidad completa sin esfuerzo.

Permisos granulares

isAdmin(member, client) para admin global. isModalityAdmin(member, client, settings) para admin de modalidad. DENIED_EMBED como respuesta reutilizable. Los permisos nunca se almacenan en DB — siempre en tiempo real.
